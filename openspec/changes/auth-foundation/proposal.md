## Why

`gastos-personales` is a multi-user finance app. Every entity
(Account, Transaction, Snapshot, Category) is owned by a `user_id`.
Every API endpoint needs to identify the caller. The auth layer is
the single dependency that every other capability (accounts,
transactions, fx, snapshots, reports, ui) sits on top of. Doing it
first lets every later change assume identity is solved.

Shipping auth as a separate, isolated change also reduces the review
budget of every later change (no need to re-validate identity in each
PR) and isolates the highest-stakes code (password hashing, OAuth
linking, session lifecycle) so it gets dedicated adversarial review.

## What

A self-contained identity subsystem built on **Auth.js v5**
(`next-auth@beta`) with the **`@auth/prisma-adapter`** so that
sessions, accounts, and verification tokens live in Postgres. The
auth subsystem supports two registration methods and one login
mechanism per user, plus one Hono-mounted application catch-all for
the non-auth API.

| Concern | Responsibility |
|---|---|
| Sessions, CSRF, OAuth flow, account linking | **Auth.js v5** (`/api/auth/*`) |
| Session storage | **Postgres** via the Prisma adapter (database sessions, not JWT) |
| Email + password registration & login | **Credentials provider** wrapping Argon2id in our code |
| Google OAuth 2.0 | **Google provider** configured in Auth.js |
| Application API (the non-auth surface) | **Hono** mounted at `/api/[...path]/route.ts` |
| Schema & migrations | **Prisma 6** + `@prisma/client` |
| Validation | **Zod** at every action boundary and in the env schema |

The two identity methods (local password, Google OAuth) are unified
at the `User` entity: a user has zero or one `passwordHash` and zero
or more `Account` rows (one per linked provider). A user can log in
through any method that has a credential registered.

### Data model

The Prisma schema is the single source of truth. The base
`User` / `Account` / `Session` / `VerificationToken` models follow
the **canonical Auth.js Prisma adapter schema**
(<https://authjs.dev/reference/adapter/prisma>). We add two columns
to `User` that the adapter does not ship. Auth.js-owned tables
(fields, types, relations) are **not** modified.

```prisma
// prisma/schema.prisma

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

model User {
  id              String    @id @default(cuid())
  name            String?
  email           String    @unique
  emailVerified   DateTime?
  image           String?

  // -- Added on top of the Auth.js canonical schema --
  passwordHash    String?   // Argon2id hash; null for OAuth-only users
  defaultProvider String    @default("local") // "local" | "google"
  lastLoginAt     DateTime? // populated by the Auth.js `signIn` callback

  accounts        Account[]
  sessions        Session[]

  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt
}

model Account {
  id                String  @id @default(cuid())
  userId            String
  type              String
  provider          String
  providerAccountId String
  refresh_token     String? @db.Text
  access_token      String? @db.Text
  expires_at        Int?
  token_type        String?
  scope             String?
  id_token          String? @db.Text
  session_state     String?

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([provider, providerAccountId])
}

model Session {
  id           String   @id @default(cuid())
  sessionToken String   @unique
  userId       String
  expires      DateTime
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}

model VerificationToken {
  identifier String
  token      String   @unique
  expires    DateTime

  @@unique([identifier, token])
}
```

`User` indexes (declared in the next migration if not implicit):

- `@@index([email])` — implicit via `@unique`, but called out for the
  reviewer.
- `@@index([createdAt])` — for the `user-deletion` and admin tools.

`Account` indexes:

- `@@unique([provider, providerAccountId])` — already declared; this
  is the only way we say "this Google account is already linked".

`Session` indexes:

- `@@unique([sessionToken])` — already declared.
- The `expires` column is scanned on every `auth()` call; an
  additional `@@index([expires])` will be added in `design` to keep
  the lookup plan clean.

`passwordHash` is **nullable** because OAuth-only users never have a
local password; the `Credentials` provider treats `null` as "this
account cannot use the email + password method" (BR-AUTH-9).

### Token strategy

The app uses **database sessions**, not application-issued JWTs.

- **Auth.js session cookie** (`authjs.session-token`, HTTP-only,
  `Secure` in production, `SameSite=Lax`). The cookie holds an
  opaque session token; the server resolves it against the `Session`
  table on every request via `auth()`.
- **Session lifetime**: 30 days (Auth.js default for database
  sessions; configurable in the Auth.js config). Sliding window:
  Auth.js extends the expiry on each request as long as the session
  has been used within the last 24 hours (BR-AUTH-7).
- **CSRF**: Auth.js double-submit pattern. All POSTs under
  `/api/auth/*` require the CSRF token from `/api/auth/csrf`. Our
  Hono application endpoints sit **outside** Auth.js's CSRF; they
  rely on the `SameSite=Lax` cookie attribute and same-origin POSTs
  from the app shell.
- **No JWTs for app sessions.** Auth.js does sign short-lived JWTs
  internally for OAuth callback handling, but the *application*
  always reads the session through `auth()`, which returns the
  database row. We never issue, verify, or store our own JWTs in
  this change.
- **No app-issued refresh tokens.** The `Session` row is the only
  thing we mint, store, and revoke. "Refresh" is the user re-using
  the cookie while it is still valid; once it expires, the user
  signs in again.

### Auth flow

**Google OAuth (Auth.js Google provider).** We configure the
provider with `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET`, scope
`openid email profile`, and `authorization.params.prompt =
"select_account"`. The full flow is owned by Auth.js:

1. UI POSTs to `/api/auth/signin/google`.
2. Auth.js redirects the browser to Google's consent screen.
3. Google redirects back to `/api/auth/callback/google?code=…`.
4. Auth.js exchanges the code, fetches the userinfo, and calls the
   `signIn` callback.
5. The Prisma adapter either creates a new `User` + `Account` row
   (no email match) or links a new `Account` row to the existing
   `User` (email match → auto-link, BR-AUTH-5). Our app code
   contributes the `signIn` callback, which stamps `lastLoginAt`
   and ensures `defaultProvider` is set on first registration.
6. Auth.js sets the session cookie. The user is signed in.

**Credentials (email + password).** The flow is one HTTP call:

1. UI POSTs to `/api/auth/callback/credentials` with
   `{ email, password }` and the CSRF token.
2. Auth.js calls the provider's `authorize(credentials, request)`
   function (our code, in `modules/auth/infrastructure/authjs.ts`).
3. The function:
   - Normalizes the email (lowercase, trim).
   - Looks up the `User` by email.
   - If not found **or** `passwordHash` is null, hashes a fixed
     dummy password to equalize response time (BR-AUTH-4), then
     returns `null`.
   - Otherwise, verifies the supplied password with Argon2id
     against `passwordHash`. On failure, returns `null`.
   - On success, returns the `User` object (only the fields
     Auth.js knows about: `id`, `email`, `name`, `image`).
4. Auth.js creates a `Session` row, sets the cookie, returns
   `200`. The UI redirects.

**Session reading.** Both server components and Hono route
handlers call `auth()` from `next-auth` v5 (the unified server-side
helper). The helper returns `{ session, user }` if the cookie is
valid, or `null` if not. The `session.user.id` is the
authorization key used by every later capability.

## Out of scope (this change)

- Other OAuth providers (Apple, Facebook, GitHub). The Prisma
  schema already supports N providers per user; only Google ships.
- Password reset / email verification flow. For MVP, password
  reset is a manual SQL update by the operator. Email verification
  is a `VerificationToken`-backed flow in a later change.
- Multi-factor auth.
- Rate limiting on the Credentials callback. The `security-rate-limiting`
  change owns it. Documented as accepted risk in BR-AUTH-12.
- Session listing / "log out all devices". Out of scope. Sign-out
  revokes only the current session (BR-AUTH-8).
- Generic ACL on top of `user_id`. Every later change handles its
  own `WHERE user_id = ?` discipline.
- UI screens (login form, register form, OAuth button). Owned by
  the `ui-auth-shell` change. The contract here is the HTTP API
  only.
- Account linking from the user profile ("Link Google to my
  account"). The flow exists on first OAuth login, but a manual
  link/unlink UI is a separate change.
- User deletion / GDPR workflows. Owned by the `user-deletion` change.

## Non-goals

- **Not building an auth-as-a-service product.** No multi-tenant
  admin panel, no tenant provisioning, no SSO.
- **Not introducing app-issued JWTs.** Sessions are DB-backed; the
  application never signs, verifies, or stores a JWT.
- **Not implementing email verification for Google signups.** We
  trust Google's `email_verified: true` claim.
- **Not sending notification emails** when an account is auto-linked.
- **Not handling user-initiated email change.** Email is the
  canonical identifier (BR-AUTH-1) and is set at first registration.

## Users and situations

| User | Situation | Touchpoint |
|---|---|---|
| New user, local | Lands on the app, wants to track personal finance | Register form (`ui-auth-shell`) → Credentials `authorize` |
| New user, Google | Prefers one-click signup | OAuth button (`ui-auth-shell`) → `/api/auth/signin/google` |
| Returning user, local | Has account with password, comes back days later | Login form → Credentials `authorize` |
| Returning user, Google | Has account linked to Google | OAuth button → Google provider |
| Active user, mixed | Registered local, later linked Google | Either method works (auto-link, BR-AUTH-5) |
| Compromised device | Wants to invalidate sessions | Sign out from that device (BR-AUTH-8). "Sign out everywhere" is a later change. |

## Business rules

1. **BR-AUTH-1 — Email is the canonical identifier.** No usernames.
   Email is normalized (lowercased, trimmed) before storage and
   lookup. Comparison is case-insensitive.
2. **BR-AUTH-2 — Local password minimum is 10 characters.** No
   complexity rules beyond length (NIST SP 800-63B). Enforced in
   the Zod schema at the register action boundary.
3. **BR-AUTH-3 — Argon2id parameters.** Tuned to ~50-100 ms hash
   time on the Fly.io 1-CPU VM. Final params (memoryCost,
   timeCost, parallelism) are decided in `design`; the proposal
   v1 baseline is `memoryCost=19456`, `timeCost=2`,
   `parallelism=1` and will be re-benchmarked in `design`.
4. **BR-AUTH-4 — User enumeration is mitigated in the Credentials
   `authorize()` function.** When the email is not found, or the
   user has no `passwordHash`, the function hashes a fixed dummy
   password with the same Argon2id parameters before returning
   `null`. Response time for "unknown email" and "wrong password"
   is statistically indistinguishable.
5. **BR-AUTH-5 — Auto-link on email match.** When Google returns
   an email that already exists in `User`, Auth.js creates a new
   `Account` row linked to the existing `User`. No password is
   requested. The `User` row keeps its existing data.
6. **BR-AUTH-6 — Google `email_verified: true` is trusted.**
   Auth.js's Google provider enforces this: if the claim is
   `false`, the OAuth flow fails and Auth.js returns an error.
7. **BR-AUTH-7 — Session expiry is 30 days, sliding.** Auth.js's
   default for database sessions. The expiry is extended on each
   request that finds a valid session, as long as the session has
   been used within the last 24 hours.
8. **BR-AUTH-8 — Sign out revokes only the current session.**
   Auth.js deletes the `Session` row whose `sessionToken` matches
   the cookie and clears the cookie. Other devices keep working.
   "Sign out everywhere" is out of scope for MVP.
9. **BR-AUTH-9 — Credentials lookup requires `passwordHash` set.**
   If the user was created via OAuth and never set a password,
   `passwordHash` is `null`. The `authorize()` function returns
   `null` (with dummy-hash timing equalization per BR-AUTH-4) and
   the UI surfaces "this account uses Google sign-in".
10. **BR-AUTH-10 — Account linking uniqueness.** The
    `@@unique([provider, providerAccountId])` constraint on
    `Account` prevents the same Google account from being linked
    to two users. If a malicious actor tried to link their Google
    to a victim's account, the second link fails and Auth.js
    returns a `OAuthAccountNotLinked` error.
11. **BR-AUTH-11 — No secrets, tokens, or password material in
    logs.** The Argon2id verification path, the session resolution
    path, and the Auth.js callbacks log only `userId` (or
    `null`) and the outcome. The structured-logging layer in
    `core/logging` is configured with a denylist of
    `{ password, passwordHash, sessionToken, access_token,
    refresh_token, id_token, csrfToken }`.
12. **BR-AUTH-12 — Failed login attempts are not rate-limited in
    MVP.** Documented as accepted risk. The
    `security-rate-limiting` change owns per-IP and per-account
    rate limits on `/api/auth/callback/credentials`. Mitigation in
    the meantime: BR-AUTH-4 equalizes timing so a brute-force
    attacker cannot distinguish "no user" from "user, wrong
    password" by latency.
13. **BR-AUTH-13 — `defaultProvider` is set on first registration**
    and never changed. For Credentials → `"local"`. For Google →
    `"google"`. The field is read by `GET /api/me` to render the
    "last sign-in method" hint in the UI.
14. **BR-AUTH-14 — Email normalization is irreversible.** The
    original (un-normalized) email is never stored. Google's
    `email_verified` is asserted on the normalized form. If a user
    later changes their email, a separate change owns it.

## Implications and impact

| Area | Impact |
|---|---|
| **Database** | New Postgres tables generated by the Auth.js Prisma adapter: `User`, `Account`, `Session`, `VerificationToken`. We add `passwordHash`, `defaultProvider`, `lastLoginAt` to `User`. Auth.js-owned tables are not modified by hand. Provider: `postgresql`. |
| **Migrations** | `prisma migrate dev` locally; `prisma migrate deploy` at container startup or via a release command on Fly.io (decided in `fly-deploy`). |
| **API surface** | 7 Auth.js-managed routes under `/api/auth/*` (handlers generated by the library) + 2 application routes under `/api/*` (`/api/health`, `/api/me`). The Hono catch-all is mounted at `app/api/[...path]/route.ts` and only handles `/api/me` and `/api/health` in this change. Later changes add `/api/accounts/*`, `/api/transactions/*`, etc. under the same catch-all. |
| **Auth.js v5 status** | `next-auth@beta` is the version everyone uses in 2026. Still labelled beta. Mitigation: pin the exact version, watch releases, plan to upgrade to stable when it ships. |
| **Domain layer** | New `auth` module: `User`, `Account`, `Session` entities; `AuthService` with `me()`, `signOut()`, `assertSession()`. Cross-module: emits `UserRegistered` (first registration only) and `UserSignedIn` (any successful sign-in) on the in-process event dispatcher. |
| **Application layer** | Auth actions orchestrate services + DTOs. Hono route handlers under `app/api/[...path]/route.ts` for `/api/me` and `/api/health`. |
| **Infrastructure** | `@auth/prisma-adapter`, `argon2` (or equivalent Argon2id implementation native to Node 20+), `@node-rs/argon2` is the chosen candidate; decided in `design`. Zod env schema validated at startup (env-config skill). |
| **Cross-module events** | `UserRegistered` (first registration, any method) and `UserSignedIn` (any successful sign-in). Downstream consumers can subscribe — e.g., to seed default categories. Out of scope to implement consumers in this change. |
| **UI** | None in this change. The Hono endpoints and the Auth.js routes are server-side only. UI is `ui-auth-shell`. |
| **CI / deploy** | No deploy in this change. Local tests only. New secrets when deploy lands: `AUTH_SECRET` (Auth.js's own secret used to sign cookies and CSRF tokens), `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `DATABASE_URL`. The env schema in `config/env.schema.ts` validates all four at startup. |
| **Bilingual docs** | This proposal + spec + design all mirrored in `Documents-es/openspec/...`. |
| **Package manager** | `pnpm` only. Lockfile is `pnpm-lock.yaml`. CI uses `pnpm install --frozen-lockfile` via `actions/setup-node` with `corepack` enabled. `npm` and `yarn` are forbidden. |
| **Security risk (documented)** | **Auto-link on email match** means whoever controls an email effectively controls the account (BR-AUTH-5). Industry-standard behaviour (Notion, Linear, Vercel). Mitigation deferred to a hardening pass. |

| **Neon cold start** | First connection on the free tier can be ~500 ms. Mitigation: warm up via a scheduled request from Fly's cron, or accept the latency in MVP. |

## Edge cases (product)

| Scenario | Behavior |
|---|---|
| Register local with already-used email | Credentials `authorize()` returns `null` (the register path is the UI's `ui-auth-shell` change; here, the contract is the auth API). UI shows "an account with this email already exists — try signing in". |
| Register local with empty email | `400 VALIDATION_ERROR` from the Zod schema. |
| Register local with 5-char password | `400 VALIDATION_ERROR` (`PASSWORD_TOO_SHORT` per BR-AUTH-2). |
| Login local with wrong password | Credentials `authorize()` returns `null` → `401` from Auth.js. |
| Login local with non-existent email | Same as wrong password: `null` + dummy-hash timing (BR-AUTH-4). |
| Login local with no `passwordHash` (Google-only user) | `null` + dummy-hash timing (BR-AUTH-9). UI surfaces "use Google sign-in". |
| OAuth callback with revoked Google token | Auth.js surfaces `OAuthSignInError`. The user lands back on the sign-in page with a generic error. |
| OAuth returns `email_verified: false` | Auth.js's Google provider rejects the flow (BR-AUTH-6). |
| OAuth returns email already linked to **same** user | Re-login. No new `User`, no new `Account`. Session row created. |
| OAuth returns email linked to a **different** user | Auto-link to the existing user (BR-AUTH-5). New `Account` row. Session row created. |
| OAuth subject already linked to a user with a **different** email | `OAuthAccountNotLinked` error. The unique constraint on `Account(provider, providerAccountId)` catches it (BR-AUTH-10). The user must sign in with the email that originally claimed the Google account. |
| User signed in with Google today, signs in with Credentials tomorrow | Works: the same `User` has both `passwordHash` (set by register-local) and an `Account` row (set by Google). Whichever method the user picks resolves to the same `userId`. |
| User signed in with Google only (never set a password), tries Credentials | `null` + dummy-hash timing (BR-AUTH-9). UI surfaces "use Google sign-in". |
| Session cookie expired | `auth()` returns `null`. UI redirects to sign-in. No 401 in the URL — the user just sees the sign-in form. |
| Session cookie valid, DB session row missing | `auth()` returns `null`. Treated as expired. The cookie is cleared on the next sign-in. |
| Google changes the user's email | The `Account` row is keyed on `providerAccountId` (Google's `sub`), not on email. The link survives. **Open question**: whether the `User.email` should be updated to the new Google email. Tracked in decision gaps. |
| Argon2 library fails to load | `500 INTERNAL_ERROR` from `/api/auth/callback/credentials`. Logged with stack, never exposed in the response. |
| DB unreachable during `authorize()` | `500 INTERNAL_ERROR`. Retried once on a fresh connection. No exponential backoff in MVP. |
| Google OAuth API down | `502 OAUTH_PROVIDER_UNAVAILABLE` from Auth.js's error mapping. UI surfaces "Google is unavailable, try again". |
| Concurrent sign-in from two devices | Each gets its own `Session` row. No contention. Sign-out on device A leaves device B's session intact (BR-AUTH-8). |
| Server restart with active sessions | Sessions live in Postgres. No session loss. Cold-start latency on first request only (see Neon row in implications). |
| User has 5+ OAuth providers in the future | Schema supports it (one `Account` row per provider per user). The `providers` field returned by Auth.js is dynamic. |

## Decision gaps (open for the next proposal/spec round)

| Question | Default if not answered | How to resolve |
|---|---|---|
| Does `User.email` get updated when Google returns a new email? | No (we keep the original). Document in `design` and surface in the UI. | Verify Auth.js v5 behavior in `design`; decide. |

## Acceptance (evidence the reviewer will see)

1. **Tests pass**: `pnpm test` (or `pnpm vitest`) exits 0. Coverage
   on the `auth` module ≥ 80% (line + branch). The Credentials
   `authorize()` and the `auth()` server-side helper are the
   minimum that must be covered.
2. **Manual smoke**: `pnpm run dev` →
   - Register local via the Credentials callback, then sign in
     with the same email + password. Session cookie is set.
   - Sign in with Google against a test OAuth client
     (`accounts.google.com` with the test client ID/secret).
   - Hit `GET /api/me` with the session cookie; response is the
     authenticated user.
   - Sign out; session cookie is cleared; `GET /api/me` returns
     `null`.
   - `GET /api/health` returns `{ status: "ok", version, uptime }`.
3. **Adversarial review**: a `reviewer` subagent audits the diff
   with focus on:
   - User enumeration in Credentials `authorize()` (BR-AUTH-4).
   - Password material in logs (BR-AUTH-11).
   - Argon2id parameter choice on the target VM (BR-AUTH-3).
   - Auto-link security model (BR-AUTH-5).
   - `Account` unique constraint and the "OAuth subject linked to
     different email" edge case (BR-AUTH-10).
   - Google `email_verified` trust (BR-AUTH-6).
   - Session expiry and sliding window (BR-AUTH-7).
   - Hono inside Next.js typing (no Next.js route-level types
     for `/api/*`).
4. **GGA**: `gga run` exits 0. Output pasted in the handoff.

## Risks (mitigated)

| Risk | Mitigation |
|---|---|
| Auth.js v5 is in beta | Pin the exact `next-auth@beta` version. Watch releases. Plan to upgrade to stable when it ships. |
| Neon cold start on free tier (~500 ms first connection) | Acceptable in MVP. Mitigation deferred to `fly-deploy`: a scheduled request warms the connection. |
| Hono mounted inside Next.js route handler means we don't get Next.js's built-in route typing for our API | Use Hono's `hc` typed client from the UI. Export an `OpenAPIHono` instance for typed consumption. |
| `pnpm` not installed in CI | `actions/setup-node` enables `corepack`, which provisions `pnpm` from `packageManager` in `package.json`. CI uses `pnpm install --frozen-lockfile`. |
| Argon2id params too slow or too fast on Fly.io 1-CPU | Benchmark in `design`. The `argon2.verify` call must complete in 50-100 ms. |
| Prisma migrations on Neon branching | `prisma migrate deploy` is idempotent and safe to run on each container startup. Final decision (startup hook vs. release command) deferred to `fly-deploy`. |
| Auto-link on email match is a security risk | Industry standard. Documented in BR-AUTH-5 and in the implications matrix. Hardening pass tracked. |
| `email_verified: false` accepted by mistake | Auth.js's Google provider enforces this; verified in design. |
| `passwordHash` accidentally logged | The structured-logging denylist (BR-AUTH-11) covers it. Lint rule forbids `console.log` in `modules/auth/`. |
